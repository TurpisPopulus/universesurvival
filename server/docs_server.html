Universe Survival Server
C# · UDP

Сервер: UDP, RAM-first, стабильный контракт

Сервер держит всю игровую симуляцию в памяти. JSON используется только как сохранение и восстановление состояния. Клиент получает актуальные данные через версии чанков и диффы.

RAM = истина

RAM-first Все данные редактируются только в памяти.

UDP протокол

Минимальный набор команд: вход, карты, объекты, позиционные пакеты.

Версии чанков

CHUNK/OBJECTS возвращают данные только при изменении версии.

Запуск

  • Старт через server/run_server.bat или dotnet run.
  • Порт по умолчанию: 7777, можно передать в аргументах запуска.
  • Данные сохраняются в server/bin/Debug/net8.0/Data (или рядом с exe).

Файлы и структура

server/
  Program.cs               # основной файл сервера
  run_server.bat           # запуск
  object_types.json        # каталог типов объектов
  resource_types.json      # каталог типов ресурсов
  blocking_types.json      # каталог типов блокировок
  surface_types.json       # каталог типов поверхностей
  accounts.json            # аккаунты (пароли + доступ)
  players.json             # состояния игроков
  Data/
    chunk_*.json            # чанки карты
    objects_*.json          # чанки объектов
    resources_*.json        # чанки ресурсов
    blocking_*.json         # чанки блокировок
    surface_*.json          # чанки поверхностей
    accounts.json
    players.json

Главный цикл и тики

  • UDP слушает входящие пакеты и обрабатывает команды.
  • Позиции рассылются по таймеру broadcastInterval (33 мс, 30 Гц).
  • Периодически сохраняются аккаунты и грязные чанки карты/объектов/ресурсов.

Рассылка идет только активным игрокам (окно активности 10 секунд).

Протокол

Все сообщения идут в защищенном конверте SEC1:

SEC1|base64(version|nonce|ciphertext|mac)

Внутри шифрованного payload передаются команды:

PING
PONG
REGISTER|name|password|appearance
LOGIN|name|password
OK
OK|x|y|accessLevel|appearance
ERR|code

id|name|x|y|appearance
CHUNK|cx|cy|lastVersion
EDIT|{...}
OBJECTS|cx|cy|lastVersion
OBJECT_EDIT|{...}
RESOURCES|cx|cy|lastVersion
RESOURCE_EDIT|{...}
BLOCKING|cx|cy|lastVersion
BLOCKING_EDIT|{...}
SURFACE|cx|cy|lastVersion
SURFACE_EDIT|{...}
  • appearance - base64 JSON с внешностью (может отсутствовать).
  • id|name|x|y|appearance - позиционный пакет, рассылается всем.
  • CHUNK и OBJECTS возвращают JSON только при росте версии.
  • OK без параметров возвращается при успешной регистрации.
  • PING/PONG - проверка доступности.

Безопасность и шифрование UDP

  • Все UDP-пакеты шифруются и подписываются: потоковый XOR на базе HMAC-SHA256 + HMAC-SHA256 (encrypt-then-MAC).
  • Пароли, appearance и координаты не уходят в открытом виде.
  • Ключ общий для клиента и сервера: server/Program.cs и universesurvival/NetworkCrypto.gd должны совпадать.
  • Сервер принимает ключ из UNIVERSE_NET_KEY (base64), иначе использует дефолт из кода.
  • Неверный MAC или версия пакета приводят к отбраковке сообщения.
  • Это защищает от пассивного перехвата, но общий ключ можно извлечь из клиента; для полноценной защиты нужен обмен ключами.

Для публичного сервера рекомендуется ключевой обмен (например, DTLS/Noise) и ротация ключей.

Антифлуд и Rate Limiting

  • Сервер применяет token-bucket на UDP-эндпоинт: по умолчанию 120 пакетов/сек, burst 200.
  • Пакеты сверх лимита отбрасываются до расшифровки, чтобы не тратить CPU.
  • Настройки читаются из окружения: UNIVERSE_RATE_LIMIT_PER_SEC и UNIVERSE_RATE_LIMIT_BURST.
  • Значение 0 для лимитов отключает защиту.
  • Состояния лимитера очищаются после 30 секунд без пакетов.

Это базовая защита от флуда; для публичных серверов используйте отдельные сетевые фильтры/DDoS-защиту.

Мониторинг производительности

  • Сервер собирает метрики производительности: CPU usage, память (RAM), количество пакетов и активных игроков.
  • Статистика логируется периодически с интервалом, настраиваемым через переменную окружения UNIVERSE_PERF_STATS_INTERVAL (по умолчанию 60 секунд).
  • Значение 0 отключает логирование метрик.
  • Формат лога: [PERF] uptime=1h 23m | cpu=2.5% | mem=45.67MB | players=5 | pkts: recv=12345 sent=23456 dropped=10

Сохранение статистики:

  • Метрики сохраняются в CSV-файлы в директории Data/perf_stats/.
  • Каждый день создается новый файл: perf_2026-01-11.csv.
  • Формат CSV: Timestamp,UptimeSeconds,CpuPercent,MemoryMB,ActivePlayers,PacketsReceived,PacketsSent,PacketsDropped
  • Сохранение в файл можно отключить через переменную окружения UNIVERSE_PERF_STATS_TO_FILE=false (по умолчанию включено).
  • CSV-файлы можно открыть в Excel, Google Sheets или анализировать скриптами для построения графиков.

Метрики:

  • uptime — время работы сервера с момента запуска.
  • cpu — процент использования CPU (среднее по всем ядрам).
  • mem — потребление оперативной памяти в мегабайтах.
  • players — количество активных игроков (исключая состояние login).
  • pkts recv — общее количество принятых UDP-пакетов.
  • pkts sent — общее количество отправленных UDP-пакетов.
  • pkts dropped — количество отброшенных пакетов из-за rate limiting.

Мониторинг помогает отслеживать нагрузку на сервер, выявлять проблемы с производительностью и определять необходимость масштабирования или оптимизации.

Аккаунты и вход

  • Аккаунты хранятся в accounts.json (хэш + соль PBKDF2).
  • При старых данных допускается миграция с plaintext пароля.
  • После LOGIN сервер возвращает позицию, accessLevel и appearance.

Имя считается занятым, если игрок активен в окне activeWindowSec.

Состояние игроков

  • Активность хранится в памяти: словарь игроков и endpoint.
  • До первого позиционного пакета игрок помечен как id="login" и не рассылается.
  • Рассылка выполняется равномерным тиком, а не по входящим пакетам.

Карта (chunks)

  • Размер чанка: 100x100 тайлов.
  • Команда CHUNK|cx|cy|lastVersion возвращает снимок при росте версии.
  • EDIT принимает массив изменений и увеличивает версию чанка.
// chunk_0_0.json
{
  "x": 0,
  "y": 0,
  "version": 12,
  "tiles": [0, 1, 2, "..."]
}

Объекты

  • Каталог типов объектов: object_types.json (typeId, displayName, isBlocking).
  • OBJECTS|cx|cy|lastVersion возвращает объекты чанка по версии.
  • OBJECT_EDIT применяет изменения и повышает версию чанка объектов.
  • Удаление объекта: typeId="__remove__".
// objects_0_0.json
{
  "x": 0,
  "y": 0,
  "version": 12,
  "objects": [
    { "entityId": "...", "typeId": "wall_wood", "x": 150, "y": 210, "rotation": 1, "isBlocking": true }
  ]
}

Ресурсы

  • Каталог типов ресурсов: resource_types.json (typeId, maxAmount, drops, gatherTool).
  • RESOURCES|cx|cy|lastVersion возвращает ресурсы чанка по версии.
  • RESOURCE_EDIT применяет изменения и повышает версию чанка ресурсов.
  • Удаление ресурса: typeId="__remove__".
  • Сервер хранит только игровые поля: typeId, amount, timestamp.
  • Визуальные параметры (texture и т.п.) сервер не использует, но хранит их в JSON для синхронизации с клиентом.
// resources_0_0.json
{
  "x": 0,
  "y": 0,
  "version": 3,
  "resources": [
    { "entityId": "...", "typeId": "tree_oak", "x": 150, "y": 210, "amount": 8 }
  ]
}

Блокировки (Blocking)

  • Каталог типов блокировок: blocking_types.json (typeId, displayName, size, color).
  • BLOCKING|cx|cy|lastVersion возвращает блокировки чанка по версии.
  • BLOCKING_EDIT применяет изменения и повышает версию чанка блокировок.
  • Удаление блокировки: typeId="__remove__".
  • Блокировки могут занимать несколько тайлов (size: [2, 2] = 2x2 тайла).
// blocking_0_0.json
{
  "x": 0,
  "y": 0,
  "version": 5,
  "entries": [
    { "entityId": "...", "typeId": "block_2x2", "x": 150, "y": 210 }
  ]
}

Настройка типов блокировок

Файл blocking_types.json описывает размеры и визуальные параметры блокировок.

{
  "types": [
    {
      "typeId": "block_1x1",
      "displayName": "Block 1x1",
      "size": [1, 1],
      "color": "#FF000080"
    },
    {
      "typeId": "block_2x2",
      "displayName": "Block 2x2",
      "size": [2, 2],
      "color": "#FF000080"
    }
  ]
}
  • typeId - уникальный id типа блокировки.
  • size - размер блокировки в тайлах [width, height].
  • color - цвет отображения в редакторе (RGBA hex).

Поверхности (Surface)

  • Каталог типов поверхностей: surface_types.json (surfaceId, name, speedMod, damage, blocking).
  • SURFACE|cx|cy|lastVersion возвращает поверхности чанка по версии.
  • SURFACE_EDIT применяет изменения и повышает версию чанка поверхностей.
  • Поверхности влияют на скорость движения, урон и проходимость.
// surface_0_0.json
{
  "x": 0,
  "y": 0,
  "version": 3,
  "surfaces": [
    { "x": 150, "y": 210, "surfaceId": 2 }
  ]
}

Настройка типов поверхностей

Файл surface_types.json описывает параметры поверхностей.

{
  "types": [
    {
      "surfaceId": 0,
      "name": "ground",
      "displayName": "Ground",
      "color": "#8B451380",
      "speedMod": 1.0,
      "damage": 0,
      "blocking": false
    },
    {
      "surfaceId": 1,
      "name": "water_shallow",
      "displayName": "Shallow Water",
      "color": "#87CEEB80",
      "speedMod": 0.7,
      "damage": 0,
      "blocking": false
    },
    {
      "surfaceId": 2,
      "name": "water_deep",
      "displayName": "Deep Water",
      "color": "#000080C0",
      "speedMod": 0.3,
      "damage": 1,
      "blocking": true
    }
  ]
}
  • surfaceId - уникальный числовой id поверхности.
  • speedMod - множитель скорости движения (1.0 = нормальная).
  • damage - урон в секунду при нахождении на поверхности.
  • blocking - блокирует ли проход (true = непроходимо).
  • color - цвет отображения в редакторе (RGBA hex).

Визуальная схема ресурсов (клиент)

  • Каждый ресурс — отдельный PNG (любой размер).
  • Базовая клетка ресурса = нижний левый тайл изображения.
  • Нижний ряд всегда непроходимый (логическая блокировка, без коллизий).
  • Верхние ряды проходимые и рисуются поверх игрока (игрок всегда за ними).
  • После добычи/удаления ресурс исчезает и проход освобождается.

Настройка типов ресурсов

Файл resource_types.json описывает масштабируемые характеристики ресурсов. Сервер использует только игровые поля, визуальные параметры игнорируются.

{
  "types": [
    {
      "typeId": "tree_oak",
      "displayName": "Oak Tree",
      "maxAmount": 8,
      "gatherTool": "axe",
      "drops": [
        { "itemId": "wood_log", "min": 2, "max": 4 }
      ],
      "texture": "res://resources/tree_oak.png"
    }
  ]
}
  • typeId - уникальный id типа.
  • maxAmount - количество ресурса в узле при создании.
  • gatherTool - чем добывать (например, axe).
  • drops - список лута с диапазоном выпадения (min/max).
  • texture - путь к PNG ресурса (нужен клиенту).

Согласование ресурсов (сервер ↔ клиент)

  • PNG с ресурсом лежит в клиенте: universesurvival/resources/*.png.
  • Один ресурс = один PNG, размер любой (клиент режет на клетки 32x32).
  • Нижний ряд PNG всегда непроходимый, верхние ряды проходимые и рисуются поверх игрока.
  • В server/resource_types.json и universesurvival/data/resource_types.json должен быть одинаковый typeId.
  • Сервер использует только игровые поля (drops/maxAmount/gatherTool); клиент использует texture.

Сохранение и атомарные записи

  • Игроки и аккаунты пишутся на диск периодически и при завершении.
  • Чанки карты, объектов и ресурсов сохраняются только при IsDirty=true.
  • Запись в JSON выполняется атомарно через временный файл.

Правило обновления документации

Любые изменения на стороне сервера должны быть отражены в этом файле и добавлены в ver_server.html с увеличением версии на 0.001.
